# アプリケーション管理システム ― 背景と目的

本書は、社内アプリケーション管理システム（以下「本システム」）を構築するにあたっての背景・問題意識・設計思想を整理した文書である。要件定義書（`02_要件定義.md`）と対をなし、「なぜそう作るのか」を後任者が理解できるよう残すことを目的とする。

---

## 1. このドキュメントについて

本システムの実装は AI コーディングエージェント（Claude Code 等）に委ねる前提だが、AI に渡す要件書だけでは判断の前提が伝わらない。仕様の「なぜ」を残すことで、将来の要件変更時に「この条件は変えてよい／変えてはいけない」の判断ができるようにする。

---

## 2. 経緯と問題意識

### 2.1 現状の課題

社内のソフトウェア資産管理は、以下のように分散している。

- **端末・インストール状況**：SKYSEA で収集されているが、管理コンソールから個別に確認する必要があり、組織横断の集計や承認状況との突合は行えていない。
- **ライセンス管理**：各部署が独自に Excel・紙台帳で管理しており、会社全体でどのソフトを何本保有しているかが把握できていない。同一製品を複数部署が別個に購入している可能性がある。
- **証書類**：購入証書・注文書・契約書がファイルサーバ上に散在しており、紛失リスクが高い。
- **フリーソフトの承認状況**：誰がどのソフトを使ってよいか、明文化された記録がない。承認のないソフトが現場でインストールされていても気付けない。
- **退職者のアカウント・ライセンス**：ユーザー単位ライセンス（特に SaaS サブスク）が退職後も残り、無駄なコストが発生している可能性がある。
- **脆弱性対応**：あるソフトに脆弱性が見つかった時、社内の誰が・どの端末でそのバージョンを使っているかを即座に把握する手段がない。

### 2.2 既存ツールでカバーされない領域

SKYSEA は「事実の収集装置」として優れているが、以下を扱わない。

- ソフトウェアの「承認状態」の管理
- ライセンスの保有契約情報（証書・本数・契約形態）
- 申請・承認ワークフロー
- 「事実 vs 権利」の突合結果の可視化
- 脆弱性アドバイザリと使用者の紐付け

本システムは SKYSEA の補完として、これらの領域を担う。

## 3. このシステムが解決したいこと

優先度順に以下を解決する。

1. **ライセンス過不足の可視化**：保有本数と実インストール数を突合し、超過・不足を検出する（最優先）。
2. **証書類の電子化と検索性確保**：紙・分散ファイルから集約し、契約単位でアクセスできるようにする。
3. **承認状態の明文化**：部署ごとに、どのソフトを許可しているかを記録し、未承認・禁止ソフトのインストールを検出する。
4. **退職者の未解除アカウント検出**：在籍状況とライセンス割当を突合し、漏れを警告する。
5. **棚卸しの記録**：年次の人間による責任確認の証跡を残す。
6. **社員からのセルフサービス的な情報アクセス**：自分のライセンス割当・自部署で承認されているソフトの検索を、社員自身ができるようにする。
7. **脆弱性発生時の影響範囲特定**（次フェーズ）：アドバイザリを登録すると、該当バージョン使用者を一覧化・通知できる基盤を用意する。

## 4. 設計思想

実装上の判断を支配する原則を以下に示す。これらは要件変更時にも保持されるべき。

### 4.1 SKYSEA との役割分担

- **SKYSEA = 事実（What is installed）**：端末ごとに何がインストールされているかを収集する。
- **本システム = 権利と承認（What should be installed / What is licensed）**：保有ライセンスと承認情報を管理し、SKYSEA からのデータと突合する。

SKYSEA のデータベースを直接参照することはサポート外操作となるため避ける。SKYSEA の自動エクスポート機能（または手動エクスポート）で生成された CSV を、共有フォルダ経由で取り込む方式を採用する。

### 4.2 ロックイン回避（FS が正本）

本システムの定着可否は読めない。使い勝手が悪ければ Excel 運用に戻る可能性も考慮する必要がある。

このため、ライセンス証書類は「**ファイルシステムを正本、データベースは検索インデックス**」という構造とする。

- 証書ファイルは人間可読なディレクトリ階層（`<ベンダー>/<製品>/<契約名>/`）に配置する。
- 各契約フォルダに `meta.yml` を自動生成し、契約情報（保有本数・期限・所管部署等）を平文テキストで残す。
- システムが廃止されても、ファイルサーバを覗くだけで全情報が読める状態を維持する。
- 全データの Excel エクスポート機能を初期から実装し、いつでも逃げられる状態を保つ。

この設計のコストとして、人がファイルを直接操作した際に DB との不整合が発生するため、整合性チェックバッチで「気付き」を提供する（ただしブロックしない）。

### 4.3 性善説に基づく運用

社員 300 人規模で同時アクセスは数人程度を想定する。利用者は社員に限定され、外部からのアクセスはない。このため：

- ライセンスキー等の機微情報も DB に平文保存可（ただしエクスポート時は明示的な選択を要求）。
- アクセス制御と監査ログで担保する。
- 暗号化・SSO 等のセキュリティ機能は MVP では入れず、必要になった段階で追加する。
- バージョン管理についても、申請が通った時点より新しいバージョンは原則 OK とする運用（個別承認を要しない）。脆弱性が見つかった時のみ、アドバイザリで「このバージョン以下は禁止」と通知する逆向きの管理にする。

### 4.4 リアルタイムが基本

データの実態反映は SKYSEA 取込み（日次）と AD 同期（日次）により常時最新化される。ダッシュボードは「**今この瞬間の状況**」を示す。

人間による責任確認（棚卸し）は内部統制の証跡として年次で別途実施する。「棚卸し」という言葉が指す 2 つの作業（データ反映と人間の責任確認）を分離する。

### 4.5 段階的な導入

MVP は最小限の機能に絞り、利用しながら必要な機能を見極めて拡張する。

- 申請ワークフロー本体、API 連携、暗号化、SSO、脆弱性アドバイザリ等は MVP では実装しない。
- ただし、将来追加できるようテーブル構造には拡張余地を持たせる（例：`approval_requests`、`product_version_advisories` テーブルの DDL のみ先行作成）。

### 4.6 全社禁止ソフトと部署裁量の両立

承認権限は各部署の情報セキュリティ管理者に移譲する。ただし、明らかに業務利用しないもの・既知の危険なソフトは全社禁止として全部署で覆せないようにする。

承認モデルは以下の 2 層構造とする。

- **全社デフォルト**（`products.default_approval_status`）：システム管理者のみ設定。
- **部署別承認**（`department_product_approvals`）：各部署の情報セキュリティ管理者が設定。

### 4.7 一般社員もログインユーザー

AD から全社員情報を同期している以上、社員もそのままシステムにログインできる。一般社員には以下の用途で参加してもらう。

- 自分に割り当てられているライセンスを確認する
- 自部署で使ってよいソフトを検索する
- 全社の製品マスタ・承認状況を閲覧する（業務効率化：他部署で承認されているソフトを参考にできる）

ダッシュボードの閲覧権限は性善説に寄せ、個別のセンシティブ情報（個人別の利用状況、退職者情報、整合性異常）以外は社員全員に開示する方針とする。

### 4.8 拡張性への配慮（Teams 等、他チャネルからの利用）

将来、Teams からの利用や Web 以外のクライアントからのアクセスが発生する可能性を考慮する。

- 通知チャネルを抽象化する（SMTP / Teams Webhook / ファイル出力 の差し替え可能）。
- HTTP ハンドラ層を「画面（templ + HTMX）」と「JSON API（将来追加）」に分離可能な構造にしておく。
- 認証層を抽象化し、Microsoft 365 連携への切替余地を残す。

ただし、これらは MVP で実装はしない。**コード上の責務分離だけ守る**。

## 5. 利害関係者と役割

| 役割                         | このシステムでの権限                                                                                   | 想定人数        |
| ---------------------------- | ------------------------------------------------------------------------------------------------------ | --------------- |
| システム管理者（情シス本部） | 全社設定・全部署横断閲覧・全社禁止登録・製品マスタ管理                                                 | 数名            |
| 部署の情報セキュリティ管理者 | 自部署の承認登録・取消、棚卸し報告                                                                     | 各部署 1 〜数名 |

## 6. スコープ

### 6.1 MVP に含めるもの

1. 製品マスタ管理（エイリアス・名寄せ・承認状態・正規ダウンロード元 URL）
2. ライセンス管理（保有本数・契約形態・満了管理・所管部署・証書添付・ライセンスキー）
3. SKYSEA CSV 取込み（フォルダ監視・名寄せキュー＋候補提示・エイリアス自動学習）
4. AD 連携（本番 LDAP・テスト CSV）
5. 部署別承認モデル（全社デフォルト＋部署別上書き、scope は部署単位）
6. ライセンス割当（ユーザー単位・端末単位）
7. ダッシュボード（過不足・未承認・禁止検出・退職者・満了・インストール推移）
8. 棚卸し（年次の責任確認記録＋完了時スナップショット）
9. サブスク管理（software_type=saas、アカウント識別子による割当）
10. 通知（SMTP メール／Teams Webhook ／ファイル出力 の切替式、満了・禁止検出・棚卸し依頼、送信履歴と失敗時の再送）
11. 設定値管理（通知再送上限・ログ保持期間等）
12. エクスポート（Excel 全件・ZIP 一括）
13. `meta.yml` の自動生成（ロックイン回避）
14. FS↔DB 整合性チェック
15. 監査ログ
16. 一般社員向けセルフサービス画面（自分のライセンス割当、全社製品検索）
17. 部署改廃時のライセンス移管 UI
18. 初期データ移行用 CLI（既存 Excel/CSV からの一括投入、検証＋ dry-run）
19. ログ・送信履歴の保持期間管理（`appmgr-prune-logs` による自動削除）
20. CLI バイナリ間の排他制御（ファイルロック）
21. テーブル DDL のみ用意するもの（画面・API は未実装）：
    - `approval_requests`（申請フロー本体用）
    - `product_version_advisories`（脆弱性アドバイザリ用）
    - `approval_scope_users` / `approval_scope_devices`（部署より細かい承認スコープ用）

### 6.2 MVP に含めないもの（将来拡張）

- 申請ワークフロー画面・申請者通知・状態遷移処理
- 脆弱性アドバイザリの登録 UI・該当者通知処理
- 各 SaaS の Admin API 連携による自動取込み（GitHub Copilot、Microsoft 365 等）
- SCIM 連携
- 部署より細かい承認スコープ（特定ユーザー・特定端末への限定）
- コストレポート・予算管理
- 証憑のバージョン管理（再発行履歴）
- 暗号化・SSO・MFA
- Teams アプリ化（Tab／メッセージ拡張など）
- 複数会社・拠点対応

## 7. 制約条件

- **動作環境**：社内 Windows Server。Linux・macOS は想定しない。
- **規模**：社員約 300 名、同時アクセス数人未満。
- **アクセス範囲**：社内ネットワーク（CDN 等の外部参照を要しない自己完結構成）。
- **テスト環境**：AD・SKYSEA から隔離されている。CSV による代替で開発・検証を行う。
- **既存システム連携**：SKYSEA（端末インベントリ）、AD（人事マスタの実体）、ファイルサーバ（証書配置先）。
- **保守体制**：開発は AI に委ね、保守は社内エンジニアが行う。Go 言語を採用し、後任者が読める範囲に留める。
- **ライセンス管理の現状**：各部署が Excel で個別管理しており、本システム導入時に統合する作業が別途発生する。

## 8. 採用しなかった選択肢と理由

設計判断の中で、検討の上で採用しなかった案を記録する。同様の議論を将来繰り返さないため。

| 採用しなかった案                                               | 理由                                                                                                                                                                                                                                               |
| -------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| SKYSEA DB 直接参照                                             | サポート外操作、スキーマ非公開、バージョン依存                                                                                                                                                                                                     |
| ファイルシステムをアプリ占有（人手禁止）                       | システム依存度が上がる、定着しなかった場合の撤退コストが大きい                                                                                                                                                                                     |
| ライセンスキーの DB 暗号化                                     | 性善説運用で十分、エクスポート時のチェックボックスで実害は防げる                                                                                                                                                                                   |
| Postgres / SQL Server                                          | 規模に対して過剰、運用負荷が増える                                                                                                                                                                                                                 |
| SPA（React 等によるフロント分離）                              | 機能規模に対して過剰、AI 生成コードの一貫性が落ちる                                                                                                                                                                                                |
| ポリモーフィックな割当テーブル統合（user/device 1 テーブル化） | SQLite は条件付き外部キー制約をサポートしないため、`target_type='user'/'device'` と `target_id` の整合性を DB レベルで保証できない。アプリ側で守る形になりバグの温床になるため、`user_assignments` と `device_assignments` の 2 テーブルに分離する |
| 承認状態を製品マスタ単一カラムで管理                           | 部署権限委譲が表現できない                                                                                                                                                                                                                         |
| 申請フローを MVP に含める                                      | 一般社員ログインは整えるが、申請の状態遷移・通知ロジックは MVP の範囲を超える。DDL のみ用意                                                                                                                                                        |
| 製品バージョン単位の承認                                       | 運用負荷が高すぎる。性善説で「申請が通った時より新しいバージョンは OK」運用とし、脆弱性は逆向きにアドバイザリで管理                                                                                                                                |
| ユーザ異動の専用履歴テーブル                                   | `audit_logs` の `users` テーブル更新ログで追跡可能。専用テーブルは MVP には不要                                                                                                                                                                    |
| CLI を単一バイナリ＋サブコマンド                               | タスクスケジューラから個別に呼び出す前提では、機能別バイナリの方が扱いやすい。`cmd/` 配下にバイナリ単位のディレクトリを分割                                                                                                                        |
| 論理削除を `is_active` (boolean) で表現                        | 「いつ無効化されたか」が失われる。退職日・廃止日・退役日は監査情報として重要なため、日時カラム（`deactivated_at` / `valid_to` / `retired_at` / `disabled_at`）で表現する。「NULL ならアクティブ」の規約で統一                                      |
| 兼務社員の複数所属を持たせるための専用テーブル                 | 社員には専任部署があり、兼務管理は本システムのスコープ外（人事システムの責務）。兼務に伴う管理権限の付与は `user_department_roles` の複数行で表現する。利用実態として別部署 PC を使うケースは `devices.primary_user_id` と `device_assignments` で十分扱える |
| 契約更新（renewal）の親子関係を DB テーブルで追跡              | 1 件の継続契約に対し新旧 2 レコードに分けて保持し、関連は `licenses.note` のテキスト参照で残す方式とする。DB 上の親子関係を持つと更新フローのバリデーションが増えるが、得られる検索性のメリットは小さい                                            |
| `audit_logs` に IP アドレスを記録                              | 社内 LAN のみで稼働し、ユーザはログイン経由で操作するため、操作主体は `app_user_id` で特定できる。IP は冗長な情報のため省略                                                                                                                       |

## 9. 用語集

| 用語                        | 定義                                                                                                           |
| --------------------------- | -------------------------------------------------------------------------------------------------------------- |
| 製品 (Product)              | ソフトウェアの正規化された単位。バージョン違い・エディション違いをまとめる場合と分ける場合がある（運用判断）。 |
| エイリアス (Product Alias)  | SKYSEA や手動登録で現れる表記揺れを正規製品に紐付けるための別名。                                              |
| ライセンス (License)        | 特定製品に対する保有契約の単位。本数・契約形態・満了日を持つ。                                                 |
| 割当 (Assignment)           | ライセンスをユーザーまたは端末に明示的に紐付ける行為。ユーザー単位ライセンスの突合根拠となる。                 |
| インストール (Installation) | SKYSEA から取り込まれた、端末上のソフトウェア検出実績。                                                        |
| 名寄せ                      | SKYSEA から取得した未知の製品表記を、正規製品マスタに紐付ける作業。                                            |
| 棚卸し (Inventory Audit)    | 部署管理者が「自部署のライセンス・承認情報を確認した」と記録する責任確認行為。年次想定。                       |
| 全社禁止                    | システム管理者が設定する、全部署で覆せない禁止状態。`globally_prohibited`。                                    |
| 部署裁量                    | 部署別承認に判断を委ねる状態。`department_discretion`。                                                        |
| 整合性チェック              | FS とDB の不整合を検出するバッチ。ブロックせず警告のみ。                                                       |
| meta.yml                    | 各契約フォルダに自動生成される、契約情報の平文テキストファイル。システム廃止時のフォールバック。               |
| アドバイザリ (Advisory)     | 製品の特定バージョン範囲に脆弱性等の問題があることを示す情報。MVP では DDL のみ用意。                          |
| 後継部署                    | 廃止された部署のライセンス・承認設定を継承する部署。`departments.successor_department_id` で表現。             |
